#### 3. Database Schema Field Name Corrections

**Problem:** TypeScript interface and SQL migration had mismatched field names causing runtime errors.

**Files Modified:** `supabase-publishing-columns.sql`

**Field Name Mapping:**
| Old Name | New Name | Type |
|----------|----------|------|
| `published` | `is_published` | BOOLEAN |
| `published_section_ids` | `publish_sections` | TEXT[] |
| `google_calendar_event_id` | `google_event_id` | TEXT |

**Before:**
```sql
ALTER TABLE wods ADD COLUMN IF NOT EXISTS published BOOLEAN DEFAULT false;
ALTER TABLE wods ADD COLUMN IF NOT EXISTS published_section_ids TEXT[];
ALTER TABLE wods ADD COLUMN IF NOT EXISTS google_calendar_event_id TEXT;
```

**After:**
```sql
ALTER TABLE wods ADD COLUMN IF NOT EXISTS is_published BOOLEAN DEFAULT false;
ALTER TABLE wods ADD COLUMN IF NOT EXISTS publish_sections TEXT[];
ALTER TABLE wods ADD COLUMN IF NOT EXISTS google_event_id TEXT;
```

**TypeScript Interface:**
```typescript
interface WOD {
  is_published: boolean
  publish_sections: string[]
  google_event_id: string | null
  publish_time: string | null
  publish_duration: number | null
}
```

**Impact:**
- Database schema now matches TypeScript interface exactly
- Eliminates runtime errors from field name mismatches

### Technical Decisions

**1. Field Name Consistency**
- Used `is_published` (boolean convention) instead of `published`
- Used `publish_sections` (stores TEXT[] section type names) instead of `published_section_ids`
- Rationale: Section type names are stable identifiers, easier to query, human-readable

**2. Optional Google Calendar**
- Made Google Calendar sync optional, not required
- Database always updates, Google sync only if credentials present
- Rationale: Reduces setup friction, prevents publishing failures due to API issues

**3. Section Filtering by Type Name**
- Filter using `section_type` (string) instead of `id` (number)
- Rationale: Section IDs could change if table recreated, type names are stable business identifiers

**4. Auto-Refresh on Publish/Unpublish**
- Automatically refresh calendar after actions
- Rationale: Immediate visual feedback, "P" badge appears/disappears instantly, better UX

**5. Athlete Logbook Filtering**
- Filter at database level, not client-side
- Rationale: More efficient, prevents exposing draft workout data to client

### Lessons Learned

1. **Revert Before Rebuild:** When inheriting broken code, reverting and rebuilding is often faster than debugging incrementally
2. **Field Name Consistency is Critical:** Mismatched field names cause hard-to-debug runtime errors
3. **Test Complete User Flows:** Individual component testing isn't enough, must test end-to-end workflows
4. **Optional Dependencies Improve Resilience:** Making external APIs optional prevents single points of failure
5. **Immediate Visual Feedback Matters:** Auto-refresh and instant badge updates provide better UX
